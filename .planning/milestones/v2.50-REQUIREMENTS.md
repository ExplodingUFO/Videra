# Requirements: v2.50 Plot Authoring Usability and Professional Chart Polish

**Defined:** 2026-04-28  
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## Milestone Intent

Tighten the single `VideraChartView` + `Plot.Add.*` chart authoring model into a more maintainable, professional, ScottPlot-style 3D charting surface.

This milestone starts from the v2.47-v2.48 chart model where `VideraChartView` is the single shipped chart control and `Plot.Add.Surface(...)`, `Plot.Add.Waterfall(...)`, and `Plot.Add.Scatter(...)` are the runtime data-loading path. It must not restore old chart views, reintroduce direct `Source` APIs, widen backend/runtime scope, or hide failures behind fallback/downshift code.

## v2.50 Requirements

### Inventory

- [x] **INV-01**: Inventory current `VideraChartView.Plot`, `Plot3D`, `Plot3DSeries`, `Plot3DAddApi`, demo, docs, tests, support evidence, and guardrails before changing public API shape.
- [x] **INV-02**: Classify chart authoring gaps as implement, document, defer, or reject.
- [x] **INV-03**: Define concise ScottPlot-style target examples for surface, waterfall, scatter, style, precision, and series lifecycle usage.
- [x] **INV-04**: Record non-goals: no old chart view compatibility, no direct `Source` API, no generic plotting engine, no backend expansion, no hidden fallback/downshift, and no god-code.

### Plot API Usability

- [x] **PLOT-01**: Users can add and inspect surface, waterfall, and scatter series through `VideraChartView.Plot.Add.*`.
- [x] **PLOT-02**: Users can reason about series identity, name, kind, and draw order without accessing internal runtime state.
- [x] **PLOT-03**: Users can clear or remove Plot-owned series through explicit Plot lifecycle methods.
- [x] **PLOT-04**: Plot lifecycle changes update revision/refresh truth deterministically and remain covered by focused tests.
- [x] **PLOT-05**: Public usage remains Plot-owned; old direct `Source` and old chart view APIs do not return.

### Professional Presentation

- [x] **PRES-01**: Users can apply professional chart presentation presets through chart-local Plot APIs.
- [x] **PRES-02**: Users can apply color-map presets without duplicating style state outside the Plot model.
- [x] **PRES-03**: Axis, legend, and probe numeric precision share one formatting policy.
- [x] **PRES-04**: Surface, waterfall, and scatter examples can report the active style/precision profile in deterministic evidence.

### Demo And Evidence

- [x] **DEMO-01**: Demo/support evidence uses only `VideraChartView.Plot.Add.*` for chart data loading.
- [x] **DEMO-02**: Demo evidence reports series count, active series identity, chart kind, color map, precision profile, and rendering status.
- [x] **DEMO-03**: Demo growth remains bounded; no broad chart workbench, editor, or hidden fallback UI is introduced.
- [x] **DEMO-04**: Focused tests cover the demo/support evidence contract.

### Guardrails And Documentation

- [x] **GUARD-01**: Repository guardrails prevent old chart view controls and direct `Source` APIs from returning.
- [x] **GUARD-02**: Package and sample documentation show the concise `VideraChartView.Plot.Add.*` usage path.
- [x] **GUARD-03**: Documentation states what is deferred: additional chart families, generic plotting engine, backend expansion, compatibility wrappers, hidden fallback/downshift behavior, and god-code.
- [x] **GUARD-04**: Public roadmap and Beads state identify v2.50 phases, dependencies, and status.

### Verification

- [x] **VER-01**: Each phase records Beads ownership, dependencies, status, and handoff notes.
- [x] **VER-02**: Implementation phases use isolated branch/worktree execution when they change code.
- [x] **VER-03**: Each phase ends with clean Beads status, branch/worktree cleanup, and pushed repository state.

## Future Requirements

- Additional chart families beyond surface, waterfall, and scatter.
- Generic plotting engine semantics beyond the current chart-local Plot model.
- Image export, publication-layout engine, or visual-regression screenshot gates.
- Backend expansion, renderer rewrite, shader/material graph, or GPU-driven chart runtime.
- Compatibility shims for removed alpha chart APIs.

## Out of Scope

| Item | Reason |
|------|--------|
| Old `SurfaceChartView` / `WaterfallChartView` / `ScatterChartView` public views | `VideraChartView` is the single shipped chart control. |
| Direct public `Source` API | `Plot.Add.*` is the public runtime data-loading path. |
| Generic plotting engine | This milestone polishes the existing 3D chart-local Plot model only. |
| Backend expansion | Professional chart usability should not widen native backend scope. |
| Hidden fallback/downshift | Unsupported or failed behavior must remain explicit diagnostics. |
| God-code workbench | Demo/support work must stay bounded and sample-first. |
