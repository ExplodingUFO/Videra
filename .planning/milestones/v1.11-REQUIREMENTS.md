# Requirements: Videra v1.11 Alpha Happy-Path Stabilization and Diagnostics Productization

**Defined:** 2026-04-18
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.11 Requirements

### Happy-Path API Stabilization

- [x] **API-07**: root README, Avalonia README, `Videra.MinimalSample`, and `Videra.ConsumerSmoke` all present the same canonical public flow based on `VideraViewOptions`, `LoadModelAsync(...)` / `LoadModelsAsync(...)`, `FrameAll()` / `ResetCamera()`, and `BackendDiagnostics`.
- [x] **API-08**: compatibility or advanced entrypoints remain supported but are clearly moved out of the default onboarding path so new alpha consumers are not asked to choose between multiple configuration stories.
- [x] **API-09**: the shortest supported first-scene path avoids `Engine`, frame hooks, pass contributors, and other extensibility concepts unless the user explicitly opts into advanced documentation.

### Diagnostics Snapshot and Reproduction Pack

- [x] **DIAG-02**: the default alpha consumer path can emit a copy-paste diagnostics snapshot that includes package/version truth, backend resolution, fallback state, platform information, and Linux display-server truth where applicable.
- [x] **DIAG-03**: support docs, issue templates, and troubleshooting guidance all consume the same diagnostics snapshot contract instead of asking users to manually assemble environment details.
- [x] **DIAG-04**: diagnostics snapshot/export works from the public viewer happy path without requiring direct `Engine` extensibility hooks or custom host-specific instrumentation.

### Alpha Feedback and Support Closure

- [x] **FB-02**: public feedback/reporting surfaces require a canonical reproduction checklist that asks whether the issue reproduces in `Videra.MinimalSample` or `Videra.ConsumerSmoke`.
- [x] **FB-03**: the alpha support flow makes it obvious which evidence is required for package-install, startup, load, camera, or diagnostics issues so first external reports are actionable.
- [x] **DOC-06**: public docs present supported paths, compatible paths, and deferred areas in one consistent alpha support story.

### Benchmark Stewardship and Release Stability

- [x] **PERF-09**: benchmark workflows preserve trend-friendly artifacts and guidance so viewer/surface-charts evidence can be compared across runs before hard thresholds are introduced.
- [x] **REL-03**: release/smoke workflows and release-facing docs validate the same canonical alpha consumer path and current public package baseline rather than divergent local-only flows.
- [x] **DOC-07**: benchmark, release, support, and onboarding docs all point to the same canonical alpha consumer path and diagnostics expectations.

## v1.12+ / Future Requirements

### Platform Expansion

- **WAYLAND-02**: native compositor-hosted Wayland embedding remains deferred until real alpha consumer feedback shows it blocks adoption more than onboarding/support friction does.

### Package Surface Expansion

- **PKG-02**: promoting `Videra.SurfaceCharts.*` to public consumer packages remains deferred until the core viewer happy path and support burden are stable under external alpha use.

### Performance Hardening

- **PERF-10**: convert benchmark stewardship from trend visibility into selective hard regression thresholds only after enough workflow history and representative consumer scenarios exist.

### Broader Product Surface

- **API-10**: additional public viewer conveniences or plugin/discovery-oriented API work remain deferred until the single happy path and diagnostics/support contract prove stable.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Promoting new viewer/runtime orchestration APIs to public surface | `VideraEngine` remains the only public extensibility root |
| Native compositor-hosted Wayland embedding | Current alpha priority is a cleaner consumer path and support loop |
| Publishing `Videra.SurfaceCharts.*` as public packages | Surface charts remain source-first for now |
| Another deep scene/runtime performance rewrite | This milestone focuses on usability and support evidence, not a new optimization loop |
| Broad demo beautification or showcase expansion | This milestone is about one canonical consumer path, diagnostics, and feedback quality |
