# Requirements: v2.27 Performance Foundation Vertical Slice

**Defined:** 2026-04-27  
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v2.27 Requirements

### Performance Diagnostics Metrics

- [x] **PDM-01**: Host can inspect viewer performance counters for draw calls, instance count, vertex count, upload bytes, resident resource count/bytes, and pickable object count when the active backend can report them.
- [x] **PDM-02**: Host can inspect SurfaceCharts visible tile count and resident tile bytes without treating SurfaceCharts as a `VideraView` mode.
- [x] **PDM-03**: Diagnostics snapshots distinguish measured metrics, estimates, unavailable values, and existing scene object counts.

### Viewer Instance Batching

- [x] **IBT-01**: Host can create a same-mesh/same-material instance batch with per-instance transform, optional color, optional object id, and a `Pickable` flag.
- [x] **IBT-02**: Invalid instance batch descriptors fail early with clear validation errors.
- [x] **IBT-03**: First-version API explicitly excludes multi-geometry batching, per-instance material, transparent instance sorting, GPU-driven culling, indirect draw, and ECS-style scene ownership.
- [x] **IBT-04**: Viewer runtime includes instance batches in framing, diagnostics, and residency/upload accounting.
- [x] **IBT-05**: Picking a pickable batch returns stable object identity plus `InstanceIndex`.
- [x] **IBT-06**: Existing non-batched object/import paths remain source-compatible and behavior-compatible.

### Benchmark and Evidence

- [x] **BME-01**: Benchmarks compare normal objects against instance batches for population/update/render-accounting cost.
- [x] **BME-02**: Benchmark evidence captures allocation, upload bytes, draw calls, resident bytes, and picking latency where available.
- [x] **BME-03**: Only stable metrics are promoted to hard benchmark thresholds; experimental metrics remain evidence-only.

### Performance Lab

- [x] **PLB-01**: Demo exposes a focused Performance Lab for normal objects versus instance batches with count/mode/pickable controls.
- [x] **PLB-02**: Performance Lab displays diagnostics for frame timing proxy, draw calls, upload bytes, resident bytes, instance count, pickable count, and pick latency where available.
- [x] **PLB-03**: Performance Lab can export support evidence for dataset settings and captured diagnostics without turning the demo into a broad showcase.

### SurfaceCharts Columnar Data

- [x] **SCD-01**: SurfaceCharts exposes a columnar scatter data entry using contiguous X/Y/Z/size/color columns.
- [x] **SCD-02**: Columnar scatter supports `ReplaceRange` and `AppendRange` with low allocation behavior and validation.
- [x] **SCD-03**: High-volume columnar scatter defaults to non-pickable and reports pickable count/state in diagnostics.
- [x] **SCD-04**: FIFO capacity and interaction-quality refine behavior are either shipped narrowly or explicitly deferred with evidence.

## Future Requirements

### Viewer Runtime Breadth

- **VRT-01**: Multi-geometry batching.
- **VRT-02**: Per-instance material overrides.
- **VRT-03**: Transparent instance sorting.
- **VRT-04**: GPU-driven culling or indirect draw.

### SurfaceCharts Breadth

- **SCH-01**: Additional chart families.
- **SCH-02**: Generic chart kernel redesign.
- **SCH-03**: Viewer/chart mode unification.

### Renderer Breadth

- **RND-01**: Animation, skeleton, morph, shadows, environment maps, post-processing, WebGL/OpenGL backend, or material node system.

## Out of Scope

| Feature | Reason |
|---|---|
| Broad architecture rewrite | The goal is a performance vertical slice, not a new engine architecture. |
| Three.js-style runtime breadth | Videra remains a .NET desktop viewer/component stack, not a general 3D engine. |
| SurfaceCharts merged into `VideraView` | Existing package and ownership boundaries stay independent. |
| New chart families | Columnar/streaming data paths should be proven before expanding chart type breadth. |
| Release publication | This milestone is product/performance foundation work, not a release cutover. |
