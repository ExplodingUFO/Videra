# Requirements: Videra v1.9 Scene Performance Evidence and Coordinator Cleanup

**Defined:** 2026-04-17
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.9 Requirements

### Telemetry and Evidence

- [x] **OBS-01**: scene runtime diagnostics expose queued bytes, uploaded bytes, upload failures, upload latency, and resolved per-frame upload budgets in addition to simple residency counts.
- [x] **OBS-02**: `RuntimeFramePrelude` and `SceneUploadQueue` surface enough telemetry to explain why a frame uploaded work, how much it uploaded, and whether budget pressure constrained that work.
- [x] **OBS-03**: a dedicated scene benchmark harness covers import, document/residency application, upload drain, and at least one representative load/rebind path.

### Upload Policy Validation

- [x] **UPLD-05**: queue drain behavior is benchmarked and tested for both normal backlog and oversized single-object cases so the current byte-budget semantics become explicit contract rather than incidental behavior.
- [x] **UPLD-06**: upload budget heuristics remain internal, but their resolved decision path is observable and regression-tested.

### Runtime Scene Coordinator Cleanup

- [x] **ARCH-03**: scene document publish, delta application, residency mutation, upload queueing, and scene diagnostics refresh move behind a dedicated internal scene-runtime coordinator rather than remaining spread across `VideraViewRuntime` partials.
- [x] **ARCH-04**: `VideraViewRuntime` stays as shell/orchestration glue for view lifecycle, options, overlay projection, and session forwarding; it should no longer be the main owner of scene mutation flow.
- [x] **ARCH-05**: scene/runtime tests reduce reliance on reflection-based access to private runtime fields by targeting the extracted coordinator and related services directly.

### API Onboarding and Discoverability

- [x] **API-03**: `VideraView` public scene/load/camera entrypoints and model-load result types carry XML docs that explain first-use behavior in IDEs.
- [x] **DOC-02**: root README, `src/Videra.Avalonia/README.md`, and `docs/extensibility.md` present one canonical quick-start flow for `VideraView`, `LoadModelAsync(...)`, `LoadModelsAsync(...)`, `FrameAll()`, `BackendDiagnostics`, and `RenderCapabilities`.
- [x] **DOC-03**: narrow samples continue to point to the same canonical public flow rather than each teaching a slightly different onboarding path.

## v1.10+ / Future Requirements

### Deeper Data-Path Optimization

- **PERF-04**: More aggressive CPU mesh release policies or upload-time payload shedding stay deferred until the new scene benchmarks show retained payloads are still the dominant memory cost.
- **NATIVE-01**: Native mesh preprocess or Rust acceleration stays deferred until benchmark evidence shows importer or mesh preparation is the next hotspot.

### Multi-Surface Productization

- **MULTI-05**: Shared-device multi-surface and offscreen productization remain deferred until the residency, budgeting, and recovery pipeline is stable.

### Upload Heuristics

- **UPLD-04**: Hardware-telemetry-driven upload autotuning remains deferred until runtime/queue-aware heuristics prove insufficient.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Promoting new viewer/runtime orchestration APIs to public surface | `VideraEngine` remains the only public extensibility root |
| Expanding `Videra.Demo` into a broader feature showcase | This milestone is about scene evidence and contract cleanup, not demo breadth |
| Full native mesh preprocess or Rust offload in the same milestone | Benchmarks should justify any deeper data-path rewrite |
| Hardware-telemetry-driven budget control | Queue/runtime telemetry is the appropriate first step |
| Large package/repository reorganization | The work stays inside scene/runtime internals and public docs |
