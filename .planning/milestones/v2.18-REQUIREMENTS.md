# Requirements

Active milestone: `v2.18 Mixed Transparent Primitive Ordering Baseline`

Mixed-transparent requirement numbering starts at `MTP-01`.

## Milestone Goal

Allow mixed `Blend` and non-`Blend` imported primitives to survive the shipped static-scene viewer/runtime path with deterministic per-primitive transparent ordering on the current renderer path, without widening into order-independent transparency, refraction/transmission, shadows, environment maps, post-processing, animation, new UI/package lines, or broader runtime/package expansion.

## Requirements

- [x] `MTP-01` Mixed transparent primitive scope lock
  - Freeze `v2.18` to mixed `Blend` / non-`Blend` imported primitive survival and deterministic per-primitive transparent ordering only on the existing viewer/runtime line.
  - Keep order-independent transparency, refraction/transmission, shadows, environment maps, post-processing, animation, new UI/package lines, backend/platform expansion, broader importer/chart widening, compatibility shims, and migration adapters out of scope.
  - Keep public package publishing and repository tag creation out of scope.

- [x] `MTP-02` Independent transparent primitive ordering contract
  - Add only the minimum renderer/runtime seam needed so the shipped static-scene path can keep mixed `Blend` and non-`Blend` imported primitives alive with deterministic transparent ordering.
  - Avoid generic transparency systems, OIT infrastructure, render-target abstraction widening, or speculative advanced-runtime abstractions.
  - Prove the new contract with focused code evidence.

- [x] `MTP-03` Repo-owned mixed-transparency proof and validation
  - Strengthen one repository-owned viewer proof path so it demonstrates the mixed-transparent primitive ordering slice on a supported host path.
  - Ensure every desktop program used in milestone validation survives `10` seconds without crashing.
  - Keep diagnostics/support artifacts coherent with the new slice and keep the proof repository-owned and non-public.

- [x] `MTP-04` Transparent primitive truth and guardrails
  - Align docs, support wording, and repository guardrails with the shipped mixed-transparent primitive ordering baseline.
  - Keep order-independent transparency, refraction/transmission, shadows, environment maps, post-processing, animation, and broader advanced-runtime breadth explicitly deferred.
  - Leave broader transparency-system decisions to later milestones.

## Out of Scope

| Item | Excluded from `v2.18` |
|---|---|
| Advanced renderer breadth | No order-independent transparency, weighted blended OIT, refraction/transmission, shadows, shadow maps, environment maps, post-processing, or fullscreen pass framework work. |
| Runtime abstraction breadth | No generic transparency-system rewrite, no new render-target/resource-set abstraction, and no wider shader-system public model. |
| Product breadth | No animation, skeletons, morph targets, mixers, new chart families, chart-kernel widening, UI-shell expansion, backend/platform expansion, or importer-format widening. |
| Compatibility | No compatibility shim, migration adapter, downgrade path, or transitional dual-path transparency contract. |
