# Requirements: Videra v1.12 Viewer-First Inspection Workflow

**Defined:** 2026-04-18
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.12 Requirements

### Section and Clipping Workflow

- [x] **CLIP-01**: host code can define, enable, disable, and clear viewer-first section / clipping planes without dropping to `Engine`-level pass extensibility APIs.
- [x] **CLIP-02**: clipping state affects the rendered scene, related overlays, and exported inspection snapshots consistently across hardware backends and software fallback.
- [x] **CLIP-03**: clipping diagnostics truth makes it obvious whether clipping is active, how many planes are in effect, and whether any fallback or limit was applied.

### Measurement and Probe Workflow

- [x] **MEAS-01**: users can create lightweight inspection measurements from the viewer surface, including at least distance and height-delta style results, without entering an editor-style manipulation mode.
- [x] **MEAS-02**: measurement anchors reuse the existing selection / annotation / world-anchor seams so overlays and scene highlighting remain internally consistent.
- [x] **MEAS-03**: host code can read, own, and clear measurement results through a stable high-level viewer contract rather than custom `Engine` hooks.

### Camera State and Snapshot Workflow

- [x] **VIEW-01**: host code can capture and restore a typed camera / view-state snapshot that includes enough information to return a user to the same inspection view.
- [x] **VIEW-02**: exported inspection snapshots include the current camera/view state plus visible inspection overlays so the saved image matches the user-visible inspection result.
- [x] **VIEW-03**: the public happy path presents `FrameAll()` / `ResetCamera()` / view-state capture and restore as part of one coherent inspection workflow.

### Alpha Inspection Productization

- [x] **DIAG-05**: diagnostics snapshot/export includes inspection workflow truth such as active clipping state, measurement counts, and snapshot/export outcome.
- [x] **DOC-08**: minimal sample, root README, Avalonia README, and troubleshooting/alpha feedback docs all explain the same inspection workflow story and current scope boundaries.
- [x] **DOC-09**: validation assets prove the inspection flow end-to-end, including clipping, measurement, camera-state restore, and snapshot export through a consumer-facing path.

## v1.13+ / Future Requirements

### Product Surface Expansion

- **INSPECT-01**: richer measurement types such as angles, polygon areas, or volume-like analysis remain deferred until the first lightweight inspection workflow proves stable.
- **EDIT-01**: editor-style clipping gizmos, transform handles, or general scene authoring remain deferred because `Videra` is a viewing/presentation stack, not a modeling tool.

### Platform Expansion

- **WAYLAND-03**: native compositor-hosted Wayland embedding remains deferred until inspection-focused alpha users prove it blocks adoption more than workflow depth does.

### Packaging Expansion

- **PKG-03**: promoting `Videra.SurfaceCharts.*` to public consumer packages remains deferred until the viewer-first inspection path and support burden are stable.

### Performance Hardening

- **PERF-11**: any new inspection-workflow performance work should be benchmark-driven after real clipping / measurement / snapshot scenarios establish a bottleneck.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Editor-style scene authoring, transform gizmos, or arbitrary object editing | `Videra` remains a viewer/presentation stack, not a modeling tool |
| Another large viewer/runtime architecture rewrite | recent milestones already stabilized the main seams; this milestone should exercise them with real user-visible workflows |
| Native compositor-hosted Wayland embedding | current ROI is lower than making inspection workflows useful and supportable |
| Publishing `Videra.SurfaceCharts.*` as public packages | this milestone stays focused on the viewer-first line |
| New deep performance pass | inspection workflows should first produce real usage and evidence before another optimization cycle |
