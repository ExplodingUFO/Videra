# Requirements: Videra v1.20 Viewer Product Boundary and Core Slimming

**Defined:** 2026-04-21
**Archived:** 2026-04-21
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.20 Requirements

### Product Boundary

- [x] **BNDR-01**: Consumer can tell from public docs and package guidance that `Videra 1.0` is a native desktop viewer / inspection / surface-chart stack, not a general-purpose game engine or Three.js-style runtime.
- [x] **BNDR-02**: Consumer can read a capability matrix that separates shipped viewer/runtime capabilities from explicitly deferred engine-`2.0` features.
- [x] **BNDR-03**: Consumer can read one package-layer matrix that consistently distinguishes `Core`, `Import`, `Backend`, `UI adapter`, and `Charts`.

### Core and Package Structure

- [x] **CORE-01**: `Videra.Core` no longer depends on concrete importer packages.
- [x] **CORE-02**: `Videra.Core` no longer depends on concrete logging provider packages.
- [x] **PKG-01**: glTF and OBJ import functionality ship through dedicated import packages that compose with `Videra.Core`.
- [x] **PKG-02**: Concrete logging providers are not implied by `Videra.Core`; runtime composition stays on logging abstractions only.

### Hosting and Validation

- [x] **HOST-01**: Host composition seams are explicit enough that runtime core, importers, backends, UI adapters, and charts can be described and consumed independently without hand-waving.
- [x] **VALD-01**: Consumer-facing docs, smoke paths, and repository guards validate the canonical slimmed package stack and fail when the new layering regresses.

## Requirement Outcomes

### Validated

- `BNDR-01` / `BNDR-02` / `BNDR-03` are validated through the new capability matrix, repository entry-doc rewrites, and passing product-boundary repository tests.
- `CORE-01` / `CORE-02` / `PKG-01` / `PKG-02` are validated through the extracted `Videra.Import.*` packages, the cleaned `Videra.Core` project file, importer unit/integration coverage, and packaged consumer smoke.
- `HOST-01` is validated through `docs/hosting-boundary.md`, reflection-based public API guards, the project-reference graph guard, and the strengthened chart-shell guard.
- `VALD-01` is validated through aligned docs, release-readiness tests, packaged consumer smoke, `Validate-Packages.ps1`, CI, and the public-release workflows.

### Deferred Future Requirements

- **RUNT-01**: User can compose scenes through stable `SceneNode`, `MeshPrimitive`, `MaterialInstance`, `Texture2D`, and `Sampler` runtime contracts.
- **RUNT-02**: The renderer exposes a stable pass vocabulary for `Opaque`, `Transparent`, `Overlay`, `Picking`, and screenshot-oriented paths.
- **GLTF-01**: Static glTF scenes render with UVs, textures, tangents, metallic-roughness, normal/emissive/alpha handling, and correct sRGB/linear behavior.
- **GLTF-02**: Material/texture upload caching and material-instance reuse are productized for static glTF scenes.
- **PERF-01**: CI fails on agreed performance regressions for first load, first frame, steady-state orbit, interaction latency, allocation budgets, and package size budgets.
- **CHRT-01**: `SurfaceCharts` ships as a first-class public product line with stable package/documentation/smoke truth.
- **CHRT-02**: A shared chart kernel is extracted only after `SurfaceCharts` proves the right reusable contracts.
- **UIAD-01**: A second UI adapter proves the new host seams outside Avalonia.
- **ENGN-01**: Advanced runtime features such as lights, shadows, environment maps, post-processing, and animation remain an opt-in `2.0` line rather than a `1.0` requirement.

### Out of Scope at Closeout

| Feature | Reason |
|---------|--------|
| Full Three.js parity | `v1.20` was about viewer/runtime product definition, not web-engine feature matching |
| `OpenGL` / `WebGL` backend work | Backend expansion still ranks below clarifying the native desktop viewer/runtime boundary |
| Physics, ECS, scripting, editor tools, or plugin marketplace | These would drag Videra toward engine/editor scope before the `1.0` viewer product is clearly defined |
| Lights, shadows, environment maps, post-processing, transparency sorting, and animation | These belong to later scene/material/runtime milestones after the core/package boundary is stable |
| Generic `Chart3D` kernel or many new 3D chart types | The milestone priority was to clarify the viewer/runtime core and package stack, not widen chart scope |
| A second UI adapter in `v1.20` | Host seams were defined here, but cross-UI validation comes later |
