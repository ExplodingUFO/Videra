# Milestone v1.9: Scene Performance Evidence and Coordinator Cleanup

**Status:** COMPLETE 2026-04-17
**Target phases:** 60-63
**Committed workstreams:** 4
**Research:** skipped — this milestone is a direct follow-on to `v1.8` and the strategy was already decided during code review

## Overview

`v1.9` 不再继续猜测 scene pipeline 的下一个瓶颈，也不把新的复杂度直接塞进 runtime。它先把 upload / residency / import 这条链补成“可测量、可解释、可回归”的系统：先加 telemetry 和 benchmark，再基于这些证据收口剩余偏厚的 scene coordinator，最后再补 public onboarding 的 XML docs 和 canonical quick-start。

这轮里程碑有一个明确边界：不继续扩大 `VideraView` public surface、不做 broader demo、不直接上 native preprocess，也不把 hardware-telemetry autotuning 硬塞进这一轮。先拿到数据，再决定更深的性能改写。

## Phase 60: Scene Upload Telemetry and Diagnostics

**Status:** COMPLETE  
**Goal:** Expose scene upload bytes, latency, budget choices, and failure information as first-class diagnostics so performance work stops depending on inference.  
**Depends on:** v1.8 complete  
**Requirements:** `OBS-01`, `OBS-02`  
**Plans:** 3 complete

**Success criteria:**
1. Scene diagnostics expose queued/uploaded bytes, upload latency, upload failures, and resolved upload budgets.
2. `RuntimeFramePrelude` preserves per-frame upload result details instead of dropping them after ready-add application.
3. `BackendDiagnostics` and narrow viewer/sample surfaces can explain current queue pressure and the last budget decision.

## Phase 61: Scene Benchmark Harness and Upload Policy Validation

**Status:** COMPLETE  
**Goal:** Add a dedicated scene benchmark harness and lock down upload-budget semantics with tests so the next performance move is evidence-driven.  
**Depends on:** Phase 60  
**Requirements:** `OBS-03`, `UPLD-05`, `UPLD-06`  
**Plans:** 3 complete

**Success criteria:**
1. BenchmarkDotNet coverage exists for import, residency apply, upload drain, and a representative load/rebind flow.
2. Oversized-single-object and normal-backlog upload semantics are covered by tests and treated as explicit contract.
3. Heuristic budget decisions are observable and benchmark output is sufficient to justify or reject deeper queue-policy changes.

## Phase 62: Scene Runtime Coordinator Extraction

**Status:** COMPLETE  
**Goal:** Extract a dedicated internal scene coordinator so `VideraViewRuntime` stops owning the full scene publish/apply/enqueue/diagnostics flow.  
**Depends on:** Phases 60-61  
**Requirements:** `ARCH-03`, `ARCH-04`, `ARCH-05`  
**Plans:** 3 complete

**Success criteria:**
1. Scene document publish, delta apply, residency mutation, upload queueing, and scene diagnostics refresh move behind a dedicated internal coordinator.
2. `VideraViewRuntime` remains the shell for view lifecycle, options, overlay, and session forwarding rather than the main scene owner.
3. Tests stop depending so heavily on reflection-based access to private runtime fields.

## Phase 63: API Onboarding and Discoverability Pass

**Status:** COMPLETE  
**Goal:** Make the stable public viewer flow obvious in IDEs and docs without widening the public API.  
**Depends on:** Phase 62  
**Requirements:** `API-03`, `DOC-02`, `DOC-03`  
**Plans:** 3 complete

**Success criteria:**
1. `VideraView` scene/load/camera entrypoints and model-load result types have XML docs.
2. Root README, Avalonia README, and extensibility docs tell one canonical public flow.
3. Narrow samples and repository guards point to the same onboarding path rather than teaching divergent ones.

## Milestone Summary

**Key decisions:**
- Start with telemetry and benchmarks before any deeper queue-policy rewrite or native data-path work.
- Treat `VideraViewRuntime` scene coordination as the remaining internal coupling hotspot, not `RenderSession`.
- Keep `VideraEngine` as the only public extensibility root and avoid widening `VideraView` API surface.
- Reserve API work for IDE/docs discoverability rather than new capabilities.
- Finish the milestone only after the new benchmark harness, scene telemetry, coordinator extraction, and onboarding guards all survive fresh verification.

**Issues deferred:**
- Hardware-telemetry-driven budget autotuning beyond benchmark-backed heuristic decisions
- Deeper CPU mesh shedding or native mesh preprocess before benchmarks justify it
- Broader multi-surface or offscreen productization beyond the current scene/runtime contract
