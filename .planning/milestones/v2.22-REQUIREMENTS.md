# Requirements: v2.22 Alpha Release Readiness Dry Run

**Defined:** 2026-04-26
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v2.22 Requirements

Requirements for the alpha release-readiness dry-run milestone. Each maps to exactly one roadmap phase.

### Clean Package Consumer Smoke

- [x] **CPS-01**: Maintainer can build the canonical public package set from `eng/public-api-contract.json` into an isolated artifact root without source-reference consumer validation.
- [x] **CPS-02**: Maintainer can run a clean packaged viewer-only consumer smoke path that uses only `Videra.Avalonia`, one platform package, and the isolated local package feed.
- [x] **CPS-03**: Maintainer can run clean packaged viewer + OBJ and viewer + glTF consumer smoke paths that use explicit `Videra.Import.Obj` and `Videra.Import.Gltf` packages and record per-scenario results.
- [x] **CPS-04**: Maintainer can run a clean packaged SurfaceCharts consumer smoke path and capture the packaged `surfacecharts-support-summary.txt` artifact.

### Doctor Evidence Packet

- [x] **DEP-01**: Maintainer can run default `Videra Doctor` and receive non-mutating text and JSON snapshots that include repository, machine, package-contract, validation-script, and support-artifact availability.
- [x] **DEP-02**: Maintainer can opt Doctor into package validation, benchmark thresholds, consumer smoke, and native validation and see `pass`, `fail`, `skip`, or `unavailable` states tied to exact scripts, prerequisites, logs, and artifacts.
- [x] **DEP-03**: Doctor evidence can correlate release dry-run, package validation, benchmark, consumer-smoke, native-validation, and demo-support artifacts without reimplementing the validators.

### Release Dry Run

- [x] **RDR-01**: Maintainer can run a read-only release dry run for a candidate version that packs the contract package set, validates package metadata and size budgets, simulates version/tag truth, and generates an evidence index.
- [x] **RDR-02**: Release dry-run artifacts expose structured status and artifact paths that Doctor, docs, and support triage can reference consistently.
- [x] **RDR-03**: Release dry run fails closed when the package contract, package build, package validation, version simulation, or evidence index is missing or stale.

### Alpha Candidate Docs

- [x] **DOC-01**: Maintainer can follow one alpha candidate checklist covering Doctor, release dry-run, package validation, Benchmark Gates, native validation, and packaged consumer smoke.
- [x] **DOC-02**: User-facing support docs say which artifact to attach for viewer, import, backend, package, native-host, and SurfaceCharts issues.
- [x] **DOC-03**: Changelog or release-note guidance describes alpha candidate validation status and known non-blockers without implying package publication, feed mutation, or tag creation.

### Blocker-Only Closeout

- [x] **BLC-01**: Maintainer can classify dry-run findings as release blockers, environment residuals, or deferred enhancements using the generated evidence.
- [x] **BLC-02**: Fixes made during closeout are narrow, directly tied to dry-run blockers, and covered by targeted tests or validation scripts; non-blocking enhancements are explicitly deferred.

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Public package publishing | This milestone is a dry run only; publishing requires a separate human-approved release milestone. |
| Public release tag creation or push | Repository release tags stay version-aligned and are not created by planning milestones. |
| Package feed mutation | Dry-run artifacts must stay local and non-mutating. |
| Public Doctor package/global tool | `Videra Doctor` remains repo-only support infrastructure. |
| Auto-remediation | Doctor reports state and routes evidence; it must not edit machine or user configuration. |
| Broad architecture rewrite | v2.22 validates the alpha candidate loop; it does not redesign Core, Import, Backend, Avalonia, or SurfaceCharts boundaries. |
| New renderer/backend/chart features | Release readiness is the goal; new product breadth would dilute validation. |
| Compatibility shims or downgrade paths | The project default is direct contracts, not compatibility layers. |
