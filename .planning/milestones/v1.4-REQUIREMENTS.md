# Requirements: Videra v1.4 Open Source Consumption and Release Surfaces

**Defined:** 2026-04-16
**Completed:** 2026-04-16
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.4 Requirements

### Public Entry Points

- [x] **ENTRY-01**: A newcomer can read `README.md` and understand what Videra is, who it is for, the current alpha/platform boundary, and the three supported starting paths: source evaluation, public package install, and contribution.
- [x] **ENTRY-02**: Public docs include a package matrix and a support matrix that clearly classify published packages, source-only modules, samples/demos, platform support, and support level.

### Public Package Distribution

- [x] **FEED-01**: Published packages install from `nuget.org` through the default documented flow, without requiring GitHub Packages credentials for the public path.
- [x] **FEED-02**: GitHub Packages, workflows, and docs explicitly describe preview/internal use only, and no public package ID is left with ambiguous dual-feed truth.

### Package Metadata & Debuggability

- [x] **PKG-01**: Public packages ship complete NuGet metadata including package icon, SourceLink, deterministic CI-friendly build metadata, and symbol-package output.
- [x] **PKG-02**: README and release surfaces expose package-version discovery and debugging-friendly package assets consistently for the public packages.

### Collaboration Intake

- [x] **COLLAB-01**: GitHub Issues, Discussions, and Security reporting are explicitly routed so usage questions, bug reports, feature requests, and vulnerabilities do not land in the same intake path.
- [x] **COLLAB-02**: Public issue templates and labels guide reporters into the correct product area (`core`, `avalonia`, `windows`, `linux`, `macos`, `surfacecharts`) and work kind (`bug`, `feature`, `docs`, `dependencies`).

### Release Surfaces

- [x] **REL-01**: Release automation produces consumer-facing notes that categorize changes and attach the intended public package artifacts.
- [x] **REL-02**: Maintainers have a documented release policy that explains versioning, feeds, asset expectations, and what is or is not publicly published.

### Maintenance Hygiene

- [x] **MAINT-01**: Dependency updates for NuGet packages and GitHub Actions run on a bounded cadence through Dependabot rather than ad hoc manual sweeps.
- [x] **MAINT-02**: Release and maintenance automation stay truthful without introducing a broad directory reorganization or silently expanding the public support contract.

## v1.5+ / Future Requirements

### Collaboration Polish

- **FORM-01**: Public issue templates can add richer form fields after the routed intake model has been exercised on real user traffic.
- **FORM-02**: Maintainers can publish a lightweight triage playbook after the first milestone of routed issues/discussions data confirms the label and intake model.

### Package Surface Expansion

- **SURF-01**: `Videra.SurfaceCharts.*` moves from source-only modules to publicly published packages only after package ownership, support policy, and release readiness are explicitly accepted.
- **SURF-02**: Additional public package feeds or pre-release channels are only introduced if they do not create ambiguous source mapping for the same public package IDs.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Large-scale repository tree reorganization | The current `src / samples / tests / docs` skeleton is already standard enough; this milestone is about public entrypoints, not moving directories for cosmetic reasons |
| Publishing `Videra.SurfaceCharts.*` as public packages immediately | The milestone first clarifies published-vs-source-only truth instead of silently promoting chart modules into a public support contract |
| New renderer/runtime/chart feature work | `v1.4` focuses on open-source consumption, release, and collaboration surfaces rather than expanding engine or chart capabilities |
| Public dual-feed distribution for the same stable package IDs | That creates source ambiguity for consumers and conflicts with the milestone goal of a single truthful public install path |
| Broad website/portal/community program work | The scope is repository-native docs, feeds, release surfaces, and GitHub collaboration metadata |
