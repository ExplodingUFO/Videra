# Requirements

Target milestone: `v2.21 Repo Doctor and Quality Gate Closure`

Requirement numbering starts at `RDG-01`.

## Milestone Goal

Improve release and support confidence by adding a repo-only `Videra Doctor` validation surface, tightening quality-gate contract truth, and adding focused allocation gates for the new importer/demo diagnostics paths delivered in `v2.20`, without widening product architecture or creating a public Doctor package.

This milestone does **not** introduce a public CLI package, auto-remediation, a broader backend API redesign, new renderer features, new UI adapters, new backend/platform families, or new SurfaceCharts/viewer integration.

## Requirements

- [x] `RDG-01` Benchmark gate contract truth
  - Align benchmark threshold repository tests, benchmark docs, and `benchmarks/benchmark-thresholds.json` with the current committed hard-gate slice.
  - Ensure the repository tests no longer encode stale threshold counts or stale benchmark names.
  - Preserve current benchmark gate semantics; do not widen thresholds to hide regressions.

- [x] `RDG-02` Repo-only Videra Doctor command
  - Add a repository-only Doctor script or command that can be run locally to summarize SDK/runtime, OS, repo state, package contract inputs, backend/platform validation availability, and existing support artifacts.
  - Keep Doctor as orchestration/reporting over existing validation surfaces rather than a new product package.
  - Emit both a human-readable summary and a structured machine-readable report under an artifacts directory.

- [x] `RDG-03` Doctor validation integration
  - Doctor can invoke or reference existing validators for package validation, benchmark threshold evaluation, consumer smoke, native validation, and demo diagnostics without reimplementing those validators.
  - Doctor output identifies skipped checks, missing prerequisites, and failed checks with actionable file/script references.
  - Doctor defaults remain non-destructive and do not publish packages, push tags, or alter user configuration.

- [x] `RDG-04` Allocation gate expansion for productized diagnostics paths
  - Add focused allocation evidence or thresholds for model import, batch import/result creation, diagnostics bundle formatting, and relevant SurfaceCharts diagnostics/reporting paths.
  - Baselines and thresholds must be source-controlled and documented with the same artifact pattern as existing benchmark gates.
  - Allocation gates should fail only on clear regressions; noisy or unstable measurements stay as evidence-only until stable.

- [x] `RDG-05` Release-readiness runbook alignment
  - Update release/readiness docs so package validation, benchmark gates, Doctor output, consumer smoke, and support artifacts describe one coherent validation sequence.
  - Keep repository-release tags and local planning milestone tags explicitly separate.
  - Ensure support docs teach when to attach Doctor output versus `Videra.Demo`, `Videra.MinimalSample`, consumer smoke, or SurfaceCharts support summaries.

- [x] `RDG-06` Repository guardrails for Doctor and gate contracts
  - Add repository tests that lock Doctor scope, artifact names, non-publishing behavior, and docs references.
  - Add tests that keep package/docs, benchmark docs, and release-readiness documentation aligned with the actual scripts and contracts.
  - Guard against Doctor becoming a public package or a god-script that owns validation logic directly.

## Out of Scope

| Item | Excluded from `v2.21` |
|---|---|
| Public Doctor package | Doctor remains repo-only and does not become a NuGet package, global tool, or supported public API. |
| Auto-remediation | Doctor reports and routes; it does not change machine configuration, package feeds, git remotes, or user settings. |
| Architecture rewrite | No Core/Import/Backend/Avalonia/SurfaceCharts reshuffle or generic 3D engine redesign. |
| Backend API redesign | No replacement of the backend abstraction or compatibility migration for advanced seams. |
| Renderer/platform breadth | No animation, skeletal/morph, lighting/shadows/post-processing, WebGL/OpenGL, new UI adapters, or new backend/platform family. |
| Publishing | No public package publication, no release-tag push, and no package feed mutation. |
