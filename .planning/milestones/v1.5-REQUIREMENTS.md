# Requirements: Videra v1.5 VideraView Runtime Thinning and Render Orchestration

**Defined:** 2026-04-17
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.5 Requirements

### Viewer Shell & Runtime

- [x] **SHELL-01**: `VideraView` exposes the same public shell surface while internal session, bridge, overlay, native-host, and input coordination move into a `VideraViewRuntime`.
- [x] **SHELL-02**: `BackendDiagnostics`, `RenderCapabilities`, `SelectionState`, and `Annotations` remain behaviorally compatible after the runtime extraction.

### Frame Scheduling

- [x] **FRAME-01**: Ready viewers render only when invalidated, instead of running a permanent `16ms` render loop while idle.
- [x] **FRAME-02**: Interactive gestures keep continuous rendering only for the lease duration, then return to single-shot rendering without degrading interaction smoothness or software presentation correctness.

### Scene Ownership & Asset Flow

- [x] **SCENE-01**: Scene ownership is unified behind a backend-neutral `SceneDocument` / `ImportedSceneAsset` path instead of split `Items` / `Engine` truth.
- [x] **SCENE-02**: Backend loss or surface rebind can rebuild scene resources from CPU-side scene assets without forcing a re-import, and multi-file import uses bounded concurrency with atomic replacement semantics.

### Interaction Core

- [x] **INPUT-01**: Camera manipulation, picking, and selection-box semantics move into Core services so Avalonia input code is limited to normalized event routing.
- [x] **INPUT-02**: Default pointer feel and selection/annotation payload truth remain stable across routed and native pointer paths, and keyboard navigation becomes an explicit supported path.

### Overlay Projection & Layout

- [x] **OVERLAY-01**: 3D-to-2D overlay projection, selection outline generation, and annotation label layout move out of `VideraViewSessionBridge` into Core overlay services.
- [x] **OVERLAY-02**: Hover, selection, and annotation overlays remain visually truthful after the extraction, including render-side hover overlay state that is currently omitted.

### Graphics Abstraction v2

- [x] **GFX-01**: Internal graphics abstraction separates device and surface responsibilities through compatibility adapters without breaking `VideraView` public API.
- [x] **GFX-02**: Runtime/orchestrator code can rebind surfaces and support future shared-device or offscreen flows without promoting new public orchestration extension points.

### Engine Internal Decomposition

- [x] **ENGINE-01**: `VideraEngine` remains the only public extensibility root while its internal scene, pass, resource-lifetime, and frame-state responsibilities are decomposed into dedicated subsystems.
- [x] **ENGINE-02**: Existing pass contributors, frame hooks, pipeline snapshots, and render capability projections remain compatible after the internal decomposition.

## v1.6+ / Future Requirements

### Multi-Surface Expansion

- **MULTI-01**: Shared-device multi-surface rendering can become a first-class scenario after graphics abstraction v2 is proven stable behind the compatibility adapter.
- **MULTI-02**: Offscreen export and thumbnail rendering can be promoted beyond internal tests once device/surface separation is exercised in production viewer flows.

### Public Surface Review

- **PUB-01**: Additional public extensibility seams are only introduced if existing `VideraEngine` contributors/hooks/capabilities cannot cover real extension scenarios.
- **PUB-02**: `SceneDocument` or imported-asset types remain internal until scene-ownership stabilization proves a public contract is justified.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Making `RenderSessionOrchestrator`, `RenderSession`, or `VideraViewSessionBridge` public extension points | This milestone explicitly preserves `VideraEngine` as the only public extensibility root |
| Expanding `VideraView` with new broad public APIs | The goal is to make the shell thinner, not to widen it |
| Unrelated chart/runtime feature work | This milestone is about viewer/runtime/graphics architecture, not chart capability expansion |
| Large repository/package reorganization | The current work is internal architecture tightening, not directory or feed restructuring |
| Full native-backend rewrite in one milestone | Graphics abstraction v2 should ship through adapters first, then migrate platform packages incrementally |
