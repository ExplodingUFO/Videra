<objective>
Recover `VIEW-03` by extracting chart orchestration into `SurfaceChartRuntime` without regressing the shipped Phase 15-18 renderer, scheduler, and probe stack.

This plan is not a renderer rewrite. It creates a runtime seam that owns authoritative state transitions while leaving visual-host and render-host responsibilities in `SurfaceChartView`.
</objective>

<guardrails>
- Keep `SurfaceChartRenderHost`, native-handle wiring, and overlay rendering in `SurfaceChartView`; this plan only extracts orchestration.
- Preserve the Phase 15-18 `SurfaceViewport`-driven scheduler and probe math by routing them through `SurfaceViewState.DataWindow.ToViewport()`.
- Do not add any chart-specific API or helper surface to `VideraView`.
</guardrails>

<threat_model>
- Medium: runtime extraction can introduce dual state between `SurfaceChartView`, `SurfaceCameraController`, and the scheduler. Mitigation: `SurfaceChartRuntime` owns the authoritative `SurfaceViewState`; lower layers consume derived compatibility views only.
- Medium: moving render-host ownership into runtime would create native-handle lifecycle bugs and UI-thread violations. Mitigation: keep render-host and native-host wiring inside `SurfaceChartView`.
- Medium: probe and overlay truth can drift if they read stale `Viewport` values. Mitigation: overlay/render sync must use runtime-derived compatibility data consistently.
- Block on high: any change that breaks the shipped render-host or tile-scheduler behavior while introducing the new runtime seam.
</threat_model>
