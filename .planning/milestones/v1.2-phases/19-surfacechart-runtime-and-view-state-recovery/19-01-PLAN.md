Create the Phase-19 core contracts that recover `VIEW-01` on top of the shipped chart stack.

This plan adds `SurfaceDataWindow`, `SurfaceCameraPose`, and `SurfaceViewState` to `Videra.SurfaceCharts.Core`, but it does so in a compatibility-first shape that preserves the existing viewport-driven LOD, probe, and render-state math from Phases 15-18.
</objective>

<guardrails>
- Keep all new contracts in `Videra.SurfaceCharts.Core`; do not add Avalonia or `VideraView` dependencies.
- `SurfaceDataWindow` must stay viewport-isomorphic (`StartX`, `StartY`, `Width`, `Height`) so the shipped Phase 15-18 bridge is shallow and explicit.
- Do not remove or rename `SurfaceViewport` in this plan. It must remain available as the compatibility model used by existing scheduler/probe/render tests.
</guardrails>

<threat_model>
- Low: persisted view-state values may contain `NaN`, `Infinity`, or non-positive spans. Mitigation: all new core contracts reject non-finite values and non-positive widths/heights.
- Medium: `SurfaceViewport` and `SurfaceDataWindow` could diverge semantically. Mitigation: add exact conversion helpers and round-trip tests; keep both shapes span-based.
- Medium: future camera pose could bypass current projection settings accidentally. Mitigation: expose a single `ToProjectionSettings()` helper on `SurfaceCameraPose` and test it explicitly.
- Block on high: any dependency from these core contracts to Avalonia, `VideraView`, or host-specific runtime types.
</threat_model>
