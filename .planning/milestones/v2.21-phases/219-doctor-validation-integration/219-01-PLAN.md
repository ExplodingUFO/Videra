# Phase 219 Plan

## Goal

Wire Doctor to the existing validation surfaces through explicit validation state reporting and opt-in invocation switches.

## Steps

1. Extend Doctor tests to require validation entries in `doctor-report.json`.
2. Add explicit switches for package validation, benchmark thresholds, consumer smoke, and native validation.
3. Keep default validation entries in `skip` state with actionable scripts/prerequisites.
4. Report unavailable prerequisites when an opt-in validator cannot run.
5. Reference demo diagnostics artifacts without inventing a new demo validator.
6. Document statuses and run switches.
7. Verify with targeted Doctor tests, docs/repository tests, manual Doctor run, diff check, and solution build.

## Success Criteria

- Doctor output distinguishes `pass`, `fail`, `skip`, and `unavailable`.
- Each validation entry includes script/artifact/prerequisite context.
- Default Doctor execution remains non-destructive and does not publish, tag, push, or edit user configuration.
