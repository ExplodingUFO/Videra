# Requirements: Videra v1.6 Scene Pipeline Truth and Backend Surface Closure

**Defined:** 2026-04-17
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.6 Requirements

### Scene Ownership Truth

- [x] **SCENE-03**: `SceneDocument` becomes the authoritative scene owner for `VideraViewRuntime` instead of a mirror rebuilt from `Engine.SceneObjects`.
- [x] **SCENE-04**: `AddObject`, `ReplaceScene`, `ClearScene`, and `Items` mutations update the document first, then apply deterministic diffs into the engine/runtime state.

### Import / Upload Separation

- [x] **IMPORT-01**: Model import produces CPU-side `ImportedSceneAsset` or `SceneDocument` fragments without immediately uploading GPU resources.
- [x] **IMPORT-02**: Multi-file scene loading uses bounded parallel import, supports cancellation, and applies atomic replace semantics so partial failures do not corrupt the active scene.

### Asset Rehydration

- [x] **ASSET-01**: Runtime upload and re-upload use the active device/surface path without treating `SoftwareResourceFactory` fallback as the normal steady-state contract.
- [x] **ASSET-02**: Backend or surface recreation can rebuild the active scene from `SceneDocument` and imported assets without forcing a fresh import or duplicating upload truth.

### Graphics Abstraction v2 Completion

- [x] **GFX-03**: Platform backend implementations adopt device/surface responsibilities directly enough to narrow the role of `LegacyGraphicsBackendAdapter`.
- [x] **GFX-04**: Viewer diagnostics, fallback reasons, and backend rebind reporting remain truthful while backend v2 migration proceeds.

### Focused Validation Surface

- [x] **LAB-01**: A narrow scene/backend lab demonstrates batch import, atomic replace, backend switch or rebind, and fallback explanation without growing into a broad catch-all demo.
- [x] **DOC-01**: README/sample/docs/repository guards describe the new scene-pipeline truth and the lab's narrow validation role consistently.

## v1.7+ / Future Requirements

### Public Contract Promotion

- **PUB-03**: `SceneDocument` or imported-scene asset types are only promoted into public API after the internal single-source contract proves stable through a full milestone.
- **PUB-04**: New viewer-side extensibility seams remain deferred unless real integrations cannot be expressed through `VideraEngine`.

### Multi-Surface / Offscreen Expansion

- **MULTI-03**: Shared-device multi-surface rendering can be promoted after platform backends stop depending on the legacy adapter for the steady-state path.
- **MULTI-04**: Offscreen export and thumbnail rendering can move beyond validation-only tests once scene rehydration and backend v2 migration are stable.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Turning `SceneDocument` into a public API immediately | This milestone is about stabilizing the internal contract first |
| Building a broad feature-rich demo UI | The goal is to prove the new scene/backend contract with a narrow lab |
| Re-expanding `VideraView` with new public orchestration APIs | `VideraEngine` remains the only public extensibility root |
| Unrelated chart/runtime feature work | This milestone is scoped to viewer scene truth and backend-v2 completion |
| One-shot rewrite of every native backend path without compatibility staging | Platform migration should remain incremental and verifiable |
