# Roadmap: v2.50 Plot Authoring Usability and Professional Chart Polish

## Milestone Goal

Tighten the single `VideraChartView` + `Plot.Add.*` chart authoring model into a more maintainable, professional, ScottPlot-style 3D charting surface without restoring old chart APIs or widening backend/runtime scope.

## Phase Details

### Phase 351: Plot API and Professional Chart Gap Inventory

**Status:** complete  
**Bead:** Videra-z44.1  
**Goal:** Map the current `VideraChartView.Plot` API, Plot-owned runtime path, chart demo, docs, tests, support evidence, and repository guardrails before changing code.

**Success criteria:**
1. Inventory classifies Plot API gaps as implement, document, defer, or reject.
2. Target examples show concise surface, waterfall, scatter, style, precision, and lifecycle usage.
3. Non-goals explicitly reject old chart views, direct `Source`, generic plotting engine scope, backend expansion, hidden fallback/downshift, and god-code.
4. Handoff identifies disjoint implementation phases and write boundaries.

**Output:** `.planning/phases/351-plot-api-and-professional-chart-gap-inventory/SUMMARY.md`

### Phase 352: Plot Series Lifecycle and Naming Polish

**Status:** complete  
**Bead:** Videra-z44.2  
**Depends on:** Phase 351  
**Goal:** Make Plot-owned series identity, naming, ordering, clear/remove, revision, and host-facing inspection easier without exposing internal runtime state.

**Success criteria:**
1. `VideraChartView.Plot.Add.*` remains the public chart data-loading path.
2. Hosts can inspect series identity, kind, name, and draw order through public Plot truth.
3. Plot lifecycle operations update revision/refresh truth deterministically.
4. Tests prove old direct `Source` and old chart view public APIs do not return.

**Output:** `.planning/phases/352-plot-series-lifecycle-and-naming-polish/SUMMARY.md`

### Phase 353: Professional Plot Presentation and Precision Wiring

**Status:** complete  
**Bead:** Videra-z44.3  
**Depends on:** Phase 351  
**Goal:** Make professional presentation presets, color maps, and numeric precision reachable through chart-local Plot APIs consistently.

**Success criteria:**
1. Plot-owned APIs expose professional presentation without duplicating style systems.
2. Axis, legend, and probe formatting share one numeric precision policy.
3. Surface, waterfall, and scatter examples can report active style/precision evidence.
4. Tests cover style/precision semantics without hidden fallback or duplicated formatting logic.

**Output:** `.planning/phases/353-professional-plot-presentation-and-precision-wiring/SUMMARY.md`

### Phase 354: Plot Demo and Support Evidence Refresh

**Status:** complete  
**Bead:** Videra-z44.4  
**Depends on:** Phases 352 and 353  
**Goal:** Refresh the chart demo/support evidence so Plot-authored workflows are easy to inspect and attach to support reports.

**Success criteria:**
1. Demo chart data loading uses only `VideraChartView.Plot.Add.*`.
2. Evidence reports series count, active series identity, chart kind, color map, precision profile, and rendering status.
3. Demo remains bounded and does not become a broad chart editor or workbench.
4. Focused tests cover the support evidence contract.

**Output:** `.planning/phases/354-plot-demo-and-support-evidence-refresh/SUMMARY.md`

### Phase 355: Single Chart View Guardrails and Documentation

**Status:** complete  
**Bead:** Videra-z44.5  
**Depends on:** Phase 354  
**Goal:** Align docs, guardrails, public roadmap, and Beads state around the single `VideraChartView` + `Plot.Add.*` professional charting path.

**Success criteria:**
1. Repository guardrails block old chart controls and direct public `Source` APIs from returning.
2. Package/sample docs show concise Plot usage and professional presentation examples.
3. Deferred scope is explicit: additional chart families, generic plotting engine, backend expansion, compatibility wrappers, hidden fallback/downshift, and god-code.
4. Beads/public roadmap and local planning state are clean.

**Output:** `.planning/phases/355-single-chart-view-guardrails-and-documentation/SUMMARY.md`
