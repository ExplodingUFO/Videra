# Roadmap

## Active Milestone

- `v1.25 SurfaceCharts Productization and Minimal Chart Kernel`
- Status: `Phase 123 planned next`
- Focus: promote `Videra.SurfaceCharts.*` from a source-first repository family into one explicit public chart product line on the existing Avalonia path, then extract only the minimum reusable chart-kernel seam needed for future chart work

### Phase 120: SurfaceCharts Product Boundary and Package Matrix

**Goal:** define one explicit shipped SurfaceCharts package boundary, install story, and support policy before widening CI or internal chart-kernel seams.
**Depends on:** `v1.18-v1.19` chart baseline plus `v1.24` shipping-gate infrastructure
**Plans:** 3 planned

Plans:

- [x] 120-01: define the shipped SurfaceCharts package set, support level, and install story on the existing Avalonia path
- [x] 120-02: align package/release/package-matrix truth with that product boundary without widening into generic `Chart3D`
- [x] 120-03: add repository guards that lock the same package boundary and keep `Videra.SurfaceCharts.Demo` separate from the install path

### Phase 121: Packaged First-Chart Consumer Proof

**Goal:** validate the public SurfaceCharts package line through one package-consumer first-chart flow on the supported host path.
**Depends on:** Phase 120
**Plans:** 3 planned

Plans:

- [x] 121-01: add one minimal package-consumer sample or smoke path that installs the SurfaceCharts packages and renders the canonical first chart
- [x] 121-02: validate install -> render -> probe/support-summary behavior in CI on the supported host path
- [x] 121-03: align smoke artifacts and support guidance with the packaged first-chart flow

### Phase 122: Minimal Chart-Kernel Seam

**Goal:** extract only the minimum reusable chart-kernel seam needed for future chart families while keeping `SurfaceChartView` thin and avoiding generic `Chart3D`.
**Depends on:** Phase 120
**Plans:** 3 planned

Plans:

- [x] 122-01: identify and extract only clearly reusable chart contracts/runtime pieces away from `SurfaceChartView` and surface-only render details
- [x] 122-02: lock the resulting minimal chart-kernel boundary with focused tests and repository truth
- [x] 122-03: preserve the same public SurfaceCharts behavior after the seam extraction without widening product scope

### Phase 123: Release Truth and Repository Enforcement for SurfaceCharts

**Goal:** align docs, release flow, package validation, and repository guards with the final shipped SurfaceCharts product line.
**Depends on:** Phase 121, Phase 122
**Plans:** 3 planned

Plans:

- [ ] 123-01: update `README.md`, package matrix, release docs, and support docs to treat `Videra.SurfaceCharts.*` as part of the public package promise
- [ ] 123-02: extend package validation and release automation to include the SurfaceCharts package line
- [ ] 123-03: add repository truth that blocks drift between docs, package set, smoke path, and the final product boundary

## Operational Baseline

- `v1.18-v1.19` already stabilized the SurfaceCharts analytics contracts, presentation defaults, and interaction semantics on the existing Avalonia path
- the SurfaceCharts projects already carry concrete `PackageId`, readme, and description metadata, but repository truth still classifies them as source-first and outside the public package promise
- `v1.24` converted benchmark evidence, performance thresholds, package-size budgets, and packaged consumer smoke into explicit shipping gates on the viewer/runtime line
- the next highest-value gap is public chart product truth, not a generic `Chart3D` abstraction, a second chart family, or a second UI adapter
