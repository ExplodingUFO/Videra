# Requirements: Videra v1.7 Scene Pipeline Closure v1

**Defined:** 2026-04-17
**Completed:** 2026-04-17
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.7 Requirements

### Runtime Test Foundation

- [x] **TEST-01**: Add a dedicated Avalonia/runtime test surface for scene pipeline work so runtime behavior no longer depends only on demo-level validation.
- [x] **TEST-02**: Cover invalidation, `beforeRender`, and baseline scene publish/add flows with focused runtime tests before deeper refactors start.

### Asset Metrics and Deferred Readiness

- [x] **ASSET-03**: `ImportedSceneAsset` exposes bounds and approximate upload-budget metrics needed for residency planning without breaking current public construction patterns.
- [x] **ASSET-04**: `Object3D` exposes deferred/recreate readiness and approximate upload cost internally without becoming a runtime-policy object.

### Scene Contract Hardening

- [x] **SCENE-05**: `SceneDocument` carries stable internal entry identity and versioning so deltas do not depend on ad hoc object-set comparisons.
- [x] **SCENE-06**: Runtime-owned imported objects and external objects have explicit ownership semantics preserved through document mutation and removal paths.

### Incremental Scene Mutation

- [x] **ITEMS-01**: `Items` collection changes use incremental document mutation for add/remove/replace/move, with full rebuild reserved for `Reset`.

### Delta Planning and Application

- [x] **DELTA-01**: Publishing a new scene document computes an explicit previous/next delta rather than doing full ad hoc engine scans inside `VideraViewRuntime.Scene`.
- [x] **DELTA-02**: Engine application is separated into dedicated scene-application services so runtime scene code becomes orchestration-only.

### Residency and Upload Budgeting

- [x] **RES-01**: Scene residency becomes explicit (`PendingUpload`, `Resident`, `Dirty`, `Failed`, etc.) instead of being inferred from GPU buffer nullability.
- [x] **RES-02**: GPU uploads move to a budgeted frame-prelude drain path and no longer happen synchronously inside `Load`, `AddObject`, `ReplaceScene`, or `ItemsChanged` flows.

### Recovery and Diagnostics

- [x] **RECOVERY-01**: Backend or surface recovery marks retained imported/runtime-owned entries dirty and reuploads them without re-importing or rebuilding the scene document.
- [x] **DIAG-01**: Viewer diagnostics project truthful scene residency and recovery state without widening the public orchestration surface.

### Import Orchestration

- [x] **IMPORT-03**: Single and batch import orchestration moves into `SceneImportService` while preserving current bounded concurrency, ordering, and batch replacement semantics.

### Validation Surface and Docs

- [x] **LAB-02**: A narrow Scene Pipeline Lab demonstrates document version, upload queue state, residency state, and backend readiness without expanding into a broad viewer demo.
- [x] **DOC-02**: README/sample/docs/repository guards describe the new document-truth, residency, upload-queue, and recovery pipeline consistently.

## v1.8+ / Future Requirements

### Mesh Payload Efficiency

- **PERF-01**: Reduce repeated CPU mesh copies through shared payload retention only after the upload/residency contract proves stable.
- **PERF-02**: Introduce CPU mesh release policies only after reupload, picking, and wireframe requirements are measured against the stabilized runtime.

### Native Data-Path Expansion

- **NATIVE-01**: Native mesh preprocess or Rust acceleration stays deferred until benchmark evidence shows importer or mesh preparation is the next hotspot.
- **MULTI-05**: Shared-device multi-surface and offscreen productization remain deferred until the scene residency and recovery pipeline is stable.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Promoting new viewer/runtime orchestration APIs to public surface | `VideraEngine` remains the only public extensibility root |
| Expanding `Videra.Demo` into a broad feature showcase | The milestone only needs a narrow scene pipeline validation surface |
| Full native-backend rewrite or removal of every legacy compatibility path in one pass | Backend migration remains incremental and compatibility-aware |
| Early Rust/native mesh preprocess work without benchmark evidence | Runtime contract closure is higher leverage than premature native optimization |
| Large package/repository reorganization | The work is internal scene/runtime closure, not repo reshaping |
