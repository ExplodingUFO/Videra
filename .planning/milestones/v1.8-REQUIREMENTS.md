# Requirements: Videra v1.8 Scene Residency Efficiency and Mesh Payload Optimization

**Defined:** 2026-04-17
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.8 Requirements

### Event-Driven Residency

- [x] **RES-03**: `SceneResidencyRegistry` only marks imported/runtime-owned entries dirty on document changes, resource-epoch changes, or explicit rehydrate requests rather than on every frame.
- [x] **RES-04**: `RuntimeFramePrelude` drains already-dirty and already-queued residency work without performing a global dirty sweep each frame.
- [x] **RES-05**: residency and upload orchestration prevent duplicate queueing or duplicate upload of entries that are already resident or already in flight.

### Mesh Payload Efficiency

- [x] **PERF-01**: imported mesh payloads can be shared across realized scene objects without duplicating vertex/index arrays for each `Object3D`.
- [x] **PERF-02**: `Object3D` retains only the CPU mesh state needed for reupload, picking, and wireframe according to an explicit retention policy.
- [x] **PERF-03**: `SceneObjectFactory` materializes deferred objects against shared mesh payloads without changing the public import/viewer entrypoints.

### Upload Budget Coordination

- [x] **UPLD-01**: `SceneUploadQueue` drains uploads under per-frame object and byte budgets instead of bulk-uploading all pending work.
- [x] **UPLD-02**: upload budgets adapt between interactive and steady-state cadence using runtime/queue-aware heuristics rather than one static budget.
- [x] **UPLD-03**: frame-prelude upload coordination keeps GPU resource creation out of `LoadModelAsync()`, `LoadModelsAsync()`, `AddObject()`, `ReplaceScene()`, and `ItemsChanged()` call paths.

### Correctness and Recovery Validation

- [x] **TEST-03**: runtime tests prove dirty marking, queueing, and upload transitions avoid duplicate uploads while preserving resident state.
- [x] **TEST-04**: payload tests prove shared mesh semantics reduce copy count without breaking wireframe, picking, or rehydration behavior.
- [x] **TEST-05**: recovery and budget tests prove backend/device rebuild rehydrates retained scene truth without reimport and respects per-frame upload budgets.

## v1.9+ / Future Requirements

### Native Data-Path Expansion

- **NATIVE-01**: Native mesh preprocess or Rust acceleration stays deferred until benchmark evidence shows importer or mesh preparation is the next hotspot.

### Multi-Surface Productization

- **MULTI-05**: Shared-device multi-surface and offscreen productization remain deferred until the residency, budgeting, and recovery pipeline is stable.

### Upload Heuristics

- **UPLD-04**: Hardware-telemetry-driven upload autotuning remains deferred until runtime/queue-aware heuristics prove insufficient.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Promoting new viewer/runtime orchestration APIs to public surface | `VideraEngine` remains the only public extensibility root |
| Expanding `Videra.Demo` into a broader feature showcase | This milestone is about runtime efficiency and correctness, not demo breadth |
| Full native mesh preprocess or Rust offload in the same milestone | The memory/residency model should stabilize before native acceleration work |
| Hardware-telemetry-driven budget control | Heuristic/runtime-mode budgeting is the appropriate first step |
| Large package/repository reorganization | The work stays inside scene/runtime internals |
