# Roadmap

## Target Milestone

`v2.27 Performance Foundation Vertical Slice`

Goal: prove Videra can render and inspect large static 3D datasets in Avalonia desktop apps with measurable performance truth, using narrow diagnostics, viewer instancing, benchmark evidence, a focused Performance Lab, and the first SurfaceCharts columnar/streaming data path.

## Scope Guard

- No broad architecture rewrite, ECS conversion, or general-purpose engine pivot.
- No animation, skeleton, morph, shadows, environment maps, post-processing, WebGL/OpenGL backend, or shader/material node system.
- No merging SurfaceCharts into `VideraView`.
- No new chart family breadth before columnar/streaming performance contracts exist.
- No release tags, public package publication, feed mutation, or release workflow execution.
- Keep changes data-oriented, measurable, and narrowly tied to performance truth.
- Prefer evidence-only metrics before hard gates unless the measurement is stable enough to block PRs.

### Phase 241: Performance Diagnostics Contract

**Status:** completed  
**Depends on:** `v2.27 milestone start`  
**Requirement:** `PDM-01`, `PDM-02`, `PDM-03`

Extend viewer and SurfaceCharts diagnostics with performance counters that distinguish scene object counts from renderer cost.

Success criteria:
1. Viewer diagnostics can report draw calls, instance count, vertex count, upload bytes, resident resource count/bytes, and pickable object count where the current backend path can measure them.
2. SurfaceCharts rendering status can report visible tile count and resident tile bytes without changing chart/viewer ownership boundaries.
3. Diagnostics snapshot formatting and support docs label metrics as measured, estimated, or unavailable.
4. Existing object-count diagnostics remain source-compatible and are explicitly documented as scene counts, not draw-call metrics.

### Phase 242: Viewer Instance Batch Contract

**Status:** completed  
**Depends on:** Phase 241  
**Requirement:** `IBT-01`, `IBT-02`, `IBT-03`

Add a narrow public instance-batch contract for same mesh/material, per-instance transform/color/id, and batch-level bounds.

Success criteria:
1. Public API exposes an instance-batch descriptor/entry shape without introducing ECS, plugin discovery, or multi-geometry batching.
2. First version supports one `MeshPrimitive`, one `MaterialInstance`, per-instance `Matrix4x4`, optional color, optional object id, and `Pickable`.
3. Transparent sorting, per-instance material, indirect draw, GPU culling, and multi-geometry batching remain explicitly out of scope.
4. Invalid descriptors fail early with clear argument errors.

### Phase 243: Instance Batch Runtime and Picking Proof

**Status:** completed  
**Depends on:** Phase 242  
**Requirement:** `IBT-04`, `IBT-05`, `IBT-06`

Wire instance batches through the retained viewer runtime enough to prove lower draw-call/object overhead and inspectable instance identity.

Success criteria:
1. A batch can be added to the viewer/runtime and included in bounds/framing.
2. Diagnostics reflect batch count, instance count, draw-call contribution, and upload bytes for the supported path.
3. Picking returns stable `ObjectId` plus `InstanceIndex` for pickable batches.
4. Non-batched object paths continue to behave as before.

### Phase 244: Benchmark Evidence and Threshold Promotion

**Status:** completed  
**Depends on:** Phase 243  
**Requirement:** `BME-01`, `BME-02`, `BME-03`

Add benchmark evidence for normal objects versus instance batches and promote only stable metrics into hard gates.

Success criteria:
1. Benchmarks compare normal object population with instance-batch population at meaningful counts.
2. Evidence captures allocation, upload bytes, draw calls, resident bytes, and hit-test/pick latency where available.
3. `benchmark-contract.json` includes the new evidence surfaces.
4. Hard thresholds are added only for stable measurements; unstable metrics remain evidence-only and documented.

### Phase 245: Performance Lab Vertical Slice

**Status:** completed  
**Depends on:** Phase 244  
**Requirement:** `PLB-01`, `PLB-02`, `PLB-03`

Add one focused Performance Lab page that demonstrates the value of the performance foundation without turning the demo into a broad showcase.

Success criteria:
1. Lab supports object count selection, normal-vs-instance mode, pickable toggle, and diagnostics display.
2. Lab shows FPS/frame-time proxy, draw calls, upload bytes, resident bytes, instance count, pickable count, and pick latency where available.
3. Lab can produce a support/export snapshot for the current dataset and settings.
4. Existing demo flows remain reachable and are not reorganized into a large new app shell.

### Phase 246: SurfaceCharts Columnar Scatter and Streaming Entry

**Status:** completed  
**Depends on:** Phase 241  
**Requirement:** `SCD-01`, `SCD-02`, `SCD-03`, `SCD-04`

Add the first columnar/streaming data entry for SurfaceCharts scatter without adding new chart families.

Success criteria:
1. SurfaceCharts exposes a columnar scatter data shape using contiguous columns for X/Y/Z, size, and color.
2. API supports `ReplaceRange` and `AppendRange` with low allocation behavior and clear validation.
3. `Pickable` defaults to false for high-volume columnar paths and is reflected in diagnostics.
4. Optional FIFO capacity and interaction-quality refine behavior are documented or explicitly deferred if implementation risk is too high.

## Current Baseline

- Viewer diagnostics already expose scene object counts and feature names, but not draw-call or upload-byte truth.
- SurfaceCharts already exposes rendering status and interaction quality, but resident tile bytes and visible tile count are not first-class public diagnostics.
- Scene/runtime data already retains primitive/material truth, but first-class instance batches do not exist.
- Scatter chart data now supports point-object series plus the first columnar append/FIFO series entry; broader SurfaceCharts streaming policies remain future work.
