# Plan 192-01: Opt-In Direct Lighting Scope Lock

## Success Criteria

1. `v2.15` is explicitly limited to one opt-in direct-lighting baseline on the existing static-scene viewer/runtime line.
2. The milestone does not widen into shadows, environment maps, post-processing, animation, or a new public package line.
3. The implementation phases stay narrow: one material/render contract phase, one repo-owned proof/diagnostics phase, and one docs/guardrails phase.
4. The later proof phase includes a Windows local check that each desktop proof app used in validation can stay alive for `10` seconds without crashing.

## Assumptions

- Existing style/uniform lighting seams are the intended baseline, not a new scene-lighting runtime model.
- Vulkan parity is the most likely minimum code gap.
- Software fallback may remain unlit if docs and proof surfaces keep that boundary honest.

## Steps

1. Lock the milestone to style-driven direct lighting only.
2. Keep the code phase focused on the minimum native-path material/render gap.
3. Keep the proof phase focused on one repo-owned lighting surface plus `10`-second no-crash validation on Windows.
4. Keep the final phase focused on docs/support/repository truth only.

## Verification

- `git status --short --branch`
- manual review of milestone scope against current lighting seams and deferred-feature boundary
