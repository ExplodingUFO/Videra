# Requirements

Active milestone: `v2.14 WPF Hosted Adapter Validation`

WPF-hosted-adapter requirement numbering starts at `WPF-01`.

## Milestone Goal

On the existing viewer/runtime line, validate one second desktop host shell through the current `WPF on Windows` proof path without turning it into a second public UI package line or widening runtime/chart/backend/import scope.

## Requirements

- [x] `WPF-01` WPF-hosted-adapter scope lock
  - Freeze `v2.14` to one repo-only second-host validation slice on `WPF on Windows`.
  - Keep public-package expansion, WinUI 3 / MAUI work, renderer/material/runtime breadth, chart-family widening, backend/platform expansion, broader importer coverage, compatibility shims, and migration adapters out of scope.
  - Keep public package publishing and repository tag creation out of scope.

- [x] `WPF-02` Shared host-seam tightening for WPF
  - Tighten only the minimum host seams needed for the existing viewer/runtime line to host cleanly through the WPF proof path.
  - Avoid growing a general multi-shell abstraction layer or a second public UI package line.
  - Prove the seam reuse with focused code evidence.

- [x] `WPF-03` Repo-only WPF golden path and diagnostics parity
  - Strengthen `smoke/Videra.WpfSmoke` so it proves one bounded viewer path on Windows.
  - Keep diagnostics/support artifacts coherent with the existing Avalonia viewer line.
  - Keep the proof explicitly repository-only.

- [x] `WPF-04` WPF repository truth and guardrails
  - Align docs, support wording, and repository guardrails with `WpfSmoke` as validation/support evidence only.
  - Keep one honest Avalonia public-package story and one honest repo-only WPF validation story.
  - Leave broader desktop-shell work to future milestones.

## Out of Scope

| Item | Excluded from `v2.14` |
|---|---|
| Public UI expansion | No second public UI adapter/package line, no WinUI 3, and no MAUI work. |
| Runtime/chart/backend breadth | No renderer/material/runtime feature expansion, no new chart family, no chart-kernel widening, and no backend/platform expansion beyond the current Windows D3D11 path. |
| Import breadth | No new importer formats, no animation/skeleton/morph, and no broader static-material feature work. |
| Compatibility | No compatibility shim, migration adapter, downgrade path, or transitional dual-shell public contract. |
| Release shape | No public package publishing, no repository release tagging, and no release-candidate workflow changes. |
