# Phase 169 Plan: Candidate Version and Tag Simulation Guardrails

## Goal

Validate candidate version/tag semantics deterministically in the release dry-run path.

## Tasks

1. Add a small version/tag simulation script.
   - Accept a candidate tag and expected package version.
   - Require `v`-prefixed tag spelling.
   - Validate the extracted tag version against `Directory.Build.props`.
   - Validate public API contract package count and project paths.

2. Integrate the script with release dry run.
   - Run before packing packages.
   - Run again against `release-dry-run-summary.json` after summary generation.

3. Add repository guardrails.
   - Assert the script exists and is wired into `Invoke-ReleaseDryRun.ps1`.
   - Assert workflow stays read-only and uses the release dry-run path.
   - Assert docs/tests vocabulary preserves simulated tag/version semantics.

4. Verify.
   - Run the new script locally with the current candidate version.
   - Run focused repository tests.

## Acceptance

- Candidate tag/version validation is credential-free.
- Candidate tag/version validation creates no tag and publishes nothing.
- Release dry-run summary is validated against the package contract.
- `RCV-02` is covered.
