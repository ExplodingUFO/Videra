# Milestone v1.5: VideraView Runtime Thinning and Render Orchestration

**Status:** COMPLETE 2026-04-17
**Target phases:** 35-41
**Committed workstreams:** 7
**Research:** skipped — milestone strategy was supplied directly in the prompt

## Overview

`v1.5` 的重点不是扩 public surface，而是把 `VideraView` 从“自己协调一切的控件”继续收成真正的 public shell。当前的主要技术债有三类：viewer 内部协调职责仍堆在控件层，`RenderSession` 仍用 ready 后常驻 `16ms` loop 驱动渲染，以及 scene / overlay / graphics abstraction 还没完全收口到稳定的 runtime 和 core seams。

这个 milestone 用 7 个 phase 顺着同一条主线推进：先把 `VideraViewRuntime` 立起来，再把 frame scheduling 改成 invalidation-driven，然后统一 scene owner、Core interaction、overlay projection、graphics abstraction v2，最后只拆 `VideraEngine` 内部，不动它作为 public extensibility root 的 façade。

## Phase 35: VideraView Runtime Shell Thinning

**Status:** COMPLETE  
**Goal:** 把 `VideraView` 收成真正的 public shell，并把 session / bridge / overlay / native-host / input 协调下沉到内部 `VideraViewRuntime`。  
**Depends on:** Previous milestones shipped viewer/session/orchestrator baseline  
**Requirements:** `SHELL-01`, `SHELL-02`  
**Plans:** 3 complete

**Success criteria:**
1. `VideraView` 不再直接持有 session/bridge/native-host/overlay/input 协调状态，而是通过内部 runtime 转发公开 API。
2. `BackendDiagnostics`、`RenderCapabilities`、`SelectionState`、`Annotations` 的对外行为不回归。
3. Runtime extraction 后，后续 frame scheduling、scene、overlay 和 graphics work 可以在不继续加厚 `VideraView` 的前提下推进。

## Phase 36: Invalidation-Driven Frame Scheduling

**Status:** COMPLETE  
**Goal:** 用 invalidation-driven runtime 和 frame lease 替代 ready 后常驻 `16ms` render loop。  
**Depends on:** Phase 35  
**Requirements:** `FRAME-01`, `FRAME-02`  
**Plans:** 3 complete

**Success criteria:**
1. Idle ready viewer 不再持续轮询渲染，空闲 CPU/GPU 占用明显下降。
2. 拖动、滚轮、动画等交互场景在 lease 持续期间保持平滑连续渲染，停下后自动回到 single-shot。
3. Software presentation copy 只在 dirty frame 后发生，同时 diagnostics、fallback 和 pipeline snapshot 口径不回归。

## Phase 37: Scene Owner and Asset Pipeline Unification

**Status:** COMPLETE  
**Goal:** 把 scene owner 固定下来，并把导入链路改成 backend-neutral 的 `SceneDocument` / `ImportedSceneAsset` 流水线。  
**Depends on:** Phases 35-36  
**Requirements:** `SCENE-01`, `SCENE-02`  
**Plans:** 3 complete

**Success criteria:**
1. `Items` 与 `Engine` 不再同时承担 scene truth，runtime 只维护一个权威 scene owner。
2. 导入阶段先产出 CPU-side scene asset，再在 backend ready/runtime upload 时生成 GPU 资源。
3. Backend 丢失或 rebind 后可以从 scene asset 重建，不必要求用户重新导入，批量加载路径使用有界并发并支持原子 replace。

## Phase 38: Core Interaction, Picking, and Selection Services

**Status:** COMPLETE  
**Goal:** 把相机操控、拾取和选择盒语义移到 Core 服务，Avalonia 只保留事件路由。  
**Depends on:** Phase 37  
**Requirements:** `INPUT-01`, `INPUT-02`  
**Plans:** 3 complete

**Success criteria:**
1. 鼠标/笔/触摸/native pointer 最终统一成稳定的 gesture snapshot，再交给 Core manipulator / picking / selection services。
2. 默认 orbit/pan/zoom/selection 手感与当前行为一致，不因为 Core 化而发生用户可见回归。
3. selection/annotation payload truth 在 routed/native pointer 路径间更稳定，并且 keyboard navigation 拥有明确支持路径。

## Phase 39: Overlay Projection and Layout Service Extraction

**Status:** COMPLETE  
**Goal:** 把 overlay 的投影与布局数学从 `VideraViewSessionBridge` 抽出来，让 bridge 回到 session state / diagnostics 适配职责。  
**Depends on:** Phases 35 and 38  
**Requirements:** `OVERLAY-01`, `OVERLAY-02`  
**Plans:** 3 complete

**Success criteria:**
1. 3D→2D 投影、selection outline、annotation label layout 和 collision/culling 在 Core overlay services 中完成，而不是混在 bridge 里。
2. hover/selection/annotation overlay 视觉不回归，并补上当前 render-side hover overlay 的缺口。
3. `VideraViewSessionBridge` 最终只同步 runtime/session/options/diagnostics truth，不再兼任 scene math 层。

## Phase 40: Graphics Abstraction v2 Device/Surface Split

**Status:** COMPLETE  
**Goal:** 把内部 graphics abstraction 从单接口 backend 升级成 device/surface 分离的 v2 结构，并通过 adapter 保持兼容。  
**Depends on:** Phases 35-39  
**Requirements:** `GFX-01`, `GFX-02`  
**Plans:** 3 complete

**Success criteria:**
1. internal graphics seam 明确区分 `IGraphicsDevice`、`IRenderSurface` 和 frame-scoped context，而不是继续把设备和表面生命周期绑在一个 backend 接口里。
2. `VideraView` public API 零破坏，现有 native backends 先经由 compatibility adapter 接入，不做一次性重写。
3. 增加至少一个 offscreen or secondary-surface 方向的集成验证，证明未来 shared-device / offscreen path 有真实落脚点。

**Status:** COMPLETE

**Depends on:** Phase 40

3. pipeline snapshot、render capability projection、tests 和 docs 在内部拆分后仍讲同一套真相。

## Milestone Summary

**Key decisions:**
- `VideraEngine` 继续作为 public extensibility root，不把 session/orchestrator/bridge 升成新的 public seams。
- `VideraView` 先变薄，再做 scheduling / scene / interaction / overlay / backend 深层收口，避免控件壳层越改越厚。
- invalidation-driven scheduling 是这轮最直接的效率收益点，因此排在所有深层 graphics refactor 之前。
- graphics abstraction v2 先用 adapter 建 seam，不用一次性重写三个平台后端。

**Delivered outcomes:**
- `VideraView` 现在是 thin public shell，runtime-local 协调落到了 `VideraViewRuntime`。
- `RenderSession` 现在用 invalidation-driven scheduling 和 interactive lease 驱动渲染，ready idle path 不再常驻 `16ms` loop。
- scene/import、interaction、overlay、graphics abstraction 和 `VideraEngine` 内部分层都已经按 milestone 目标完成收口。

**Issues deferred:**
- 新的 public extensibility roots
- 与本 milestone 无关的 renderer/chart feature expansion
- 一次性重写全部 native backend 平台实现
