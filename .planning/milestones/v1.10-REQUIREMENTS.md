# Requirements: Videra v1.10 Alpha Consumer Integration and Feedback Loop

**Defined:** 2026-04-17
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v1.10 Requirements

### Happy-Path Viewer API

- [x] **API-04**: root README, Avalonia README, and the default minimal sample present one short public flow based on `VideraViewOptions`, `LoadModelAsync(...)` / `LoadModelsAsync(...)`, `FrameAll()` / `ResetCamera()`, and `BackendDiagnostics`.
- [x] **API-05**: advanced extensibility entrypoints (`Engine`, frame hooks, pass contributors) remain supported but are moved out of default getting-started examples and documented only as opt-in extensibility paths.
- [x] **API-06**: the primary minimal sample optimizes time-to-first-scene and avoids requiring new users to understand internal engine concepts before they can load and frame a model.

### Consumer Validation

- [x] **CONS-01**: a consumer-grade smoke path installs `Videra.Avalonia` plus one platform package from the public package flow and validates initialize → load model → frame camera → read diagnostics.
- [x] **CONS-02**: consumer smoke validation runs from workflow entrypoints so regressions in package install, startup, model load, or diagnostics projection are caught before release.
- [x] **CONS-03**: consumer validation and public docs keep Linux support truthful: X11-hosted native path and Wayland-session XWayland compatibility are supported alpha paths; native compositor-hosted Wayland embedding remains deferred.

### Performance Regression Gates

- [x] **PERF-05**: viewer benchmarks and existing surface-chart benchmarks are runnable from explicit workflow entrypoints rather than only by ad hoc local commands.
- [x] **PERF-06**: benchmark workflows can publish artifacts/results for manual or labeled runs so performance evidence survives beyond one machine.
- [x] **PERF-07**: benchmark gate docs clearly separate informational evidence from blocking regression signals so alpha contributors can interpret the results correctly.

### Alpha Feedback and Support Surfaces

- [x] **FB-01**: alpha-facing issue/discussion/reporting surfaces ask for the diagnostics and support data needed to reproduce consumer integration bugs.
- [x] **DOC-04**: public docs position Videra as an early alpha embeddable viewer component with explicit supported paths, deferred areas, and feedback expectations.
- [x] **DOC-05**: public alpha docs, sample surfaces, and feedback templates point to the same canonical consumer path instead of teaching divergent integration stories.

## v1.11+ / Future Requirements

### Platform Expansion

- **WAYLAND-01**: native compositor-hosted Wayland embedding remains deferred until alpha consumer feedback proves it is a blocking adoption issue rather than a lower-frequency advanced-host request.

### Package Surface Expansion

- **PKG-01**: promoting `Videra.SurfaceCharts.*` from source-first modules to public consumer packages remains deferred until the core viewer package path is smoother and better validated.

### Deeper Runtime Optimization

- **PERF-08**: deeper scene/runtime optimization beyond the new benchmark gates remains deferred until consumer smoke and external feedback show the next real bottleneck.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Promoting new viewer/runtime orchestration APIs to public surface | `VideraEngine` remains the only public extensibility root |
| Native compositor-hosted Wayland embedding | Current alpha priority is consumer path clarity and feedback, not new platform claims |
| Publishing `Videra.SurfaceCharts.*` as public packages | Surface charts remain source-first for now |
| Another deep internal scene/runtime rewrite | The next gap is adoption friction, not missing internal machinery |
| Broad demo beautification or feature expansion | This milestone is about integration, validation, and feedback loops |
