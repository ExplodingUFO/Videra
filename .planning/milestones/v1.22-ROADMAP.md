### Phase 111: UV, Texture Assets, and Color-Space Truth

**Goal:** thread static glTF UV coordinates, texture references, and explicit color-space intent through the importer/runtime path without introducing importer-shaped public types.
**Depends on:** `v1.21` scene/material runtime contracts and repository truth
**Plans:** 3 completed

Plans:

- [x] 111-01: thread static glTF UV coordinates and texture references into explicit `Texture2D` / `Sampler` runtime assets without widening public importer/backend surface area
- [x] 111-02: define one clear color-space contract for imported texture usages so base-color/emissive textures and data textures bind with truthful sRGB/linear intent
- [x] 111-03: prove imported texture assets bind through the existing viewer/runtime path without reintroducing importer-shaped shortcuts

### Phase 112: Metallic-Roughness, Alpha, Emissive, and Normal Material Semantics

**Goal:** productize the core static PBR material semantics on the shipped viewer path through explicit runtime contracts instead of backend-shaped material logic.
**Depends on:** Phase 111
**Plans:** 3 completed

Plans:

- [x] 112-01: map static glTF metallic-roughness and alpha semantics onto explicit viewer-path material contracts
- [x] 112-02: thread emissive and normal-map-ready material inputs through the same runtime material surface instead of pass-local special cases
- [x] 112-03: keep the material-semantics work scoped to the shipped static baseline rather than widening into broader advanced-runtime feature requests

### Phase 113: Tangent Inputs and Upload Reuse for Static Scenes

**Goal:** make tangent-aware shading data and material/texture upload reuse explicit enough that repeated static glTF scenes flow through one predictable runtime path.
**Depends on:** Phase 112
**Plans:** 3 completed

Plans:

- [x] 113-01: carry tangent-aware shading inputs through the importer/runtime path as explicit runtime data instead of importer-only side channels
- [x] 113-02: productize texture/material upload reuse for repeated static glTF scenes without adding compatibility layers or speculative cache abstractions
- [x] 113-03: keep the runtime path direct and maintainable so static-scene reuse does not collapse back into importer-shaped payload rebuilding

### Phase 114: Docs, Samples, and Repository Truth for Static glTF/PBR

**Goal:** make docs, samples, and repository guards teach the same static glTF/PBR baseline that the code ships, including deliberate exclusions.
**Depends on:** Phase 113
**Plans:** 3 completed

Plans:

- [x] 114-01: update README, architecture docs, and package guidance to explain the shipped static glTF/PBR baseline in viewer-first terms
- [x] 114-02: align demo/sample truth with the new static glTF/PBR path without leaking backend-specific types into public examples
- [x] 114-03: add or update repository guards so docs, samples, and code stay aligned on the same static glTF/PBR baseline and exclusions

## Operational Baseline

- `v1.21` established stable scene/material runtime objects plus one render-feature vocabulary and aligned docs/sample truth
