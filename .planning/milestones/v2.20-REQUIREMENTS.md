# Requirements Archive: v2.20 API/DX Productization and Diagnostics Contract Closure

**Archived:** 2026-04-26  
**Status:** complete  
**Audit:** passed

## Milestone Goal

Improve alpha-stage product confidence by making Videra easier to install, easier to wire for model import, easier to diagnose from demos/support artifacts, and clearer about backend capabilities, while preserving the existing Core / Import / Backend / Avalonia / SurfaceCharts boundaries.

This milestone did **not** introduce a broad architecture rewrite, new renderer feature breadth, SurfaceCharts/viewer merger, new UI adapters, new backend/platform families, animation, skeletal/morph support, lighting/shadows/post-processing, WebGL/OpenGL, or generic 3D engine parity.

## Requirements

- [x] `PDX-01` Package/docs contract consistency
  - Corrected `Videra.Import.Gltf` and `Videra.Import.Obj` README text so importer packages are described as explicit dependencies for importer-backed `LoadModelAsync(...)` / `LoadModelsAsync(...)` paths.
  - Added root README install-by-scenario guidance for viewer-only, viewer + OBJ, viewer + glTF, SurfaceCharts, and Core-only use.
  - Added lightweight package/docs contract checks that catch drift between README install guidance, package matrix truth, and `.csproj` package references.

- [x] `PDX-02` Async importer registry API
  - Introduced importer contracts that answer `CanImport(path)` and import through cancellation-aware async request/result semantics.
  - Preserved explicit `Videra.Import.*` package installation; `Videra.Avalonia` did not regain transitive importer dependencies.
  - Added low-boilerplate registration through `ModelImporters` and `UseModelImporter(...)`.

- [x] `PDX-03` Load model result and partial-failure diagnostics
  - Surfaced import diagnostics, warning/error details, and import duration through structured result models.
  - Made multi-file loading report per-file success/failure information without hiding partial success.
  - Kept existing happy paths usable while docs/samples teach the new diagnostic path.

- [x] `PDX-04` Diagnostics-first Videra.Demo support flow
  - Added copyable diagnostics bundle containing OS/runtime/backend, selected backend/fallback status, package versions where available, loaded model count, import diagnostics, upload/render state, and render capabilities.
  - Added import report surface that shows each attempted file, success/failure, parse/import timing, asset counts, and warnings/errors.
  - Added minimal reproduction metadata that captures scene path(s), relevant settings, diagnostics snapshot, and interaction state without becoming an editor/project format.

- [x] `PDX-05` Backend advanced API capability truth
  - Made advanced backend seams such as shader/resource-set creation explicitly capability-gated through interfaces and `RenderCapabilities` flags.
  - Added documentation/diagnostics for public APIs that may be unsupported by shipped backends.
  - Avoided widening backend architecture beyond the existing shipped viewer/runtime contract.

- [x] `PDX-06` SurfaceCharts demo diagnostics/fallback closure
  - Added a `SurfaceCharts.Demo` diagnostics panel that exposes `RenderingStatus`, fallback status, and relevant host/backend state.
  - Kept `SurfaceChartView` and chart semantics independent from `VideraView`.
  - Documented the demo's diagnostic boundary honestly without claiming all GPU host/fallback combinations are exhaustively showcased.

## Future Requirements

- `DOC-01` Repo-only `Videra Doctor` command or script for package/backend/native validation triage.
- `PERF-01` Additional allocation gates beyond the `v2.19` benchmark hard-gate expansion if new importer/demo diagnostics paths introduce measurable churn.
- `API-01` Broader backend API redesign or compatibility migration if capability gating exposes deeper public-contract pressure.

## Out of Scope

| Item | Excluded from `v2.20` |
|---|---|
| Architecture rewrite | No Core/Import/Backend/Avalonia/SurfaceCharts reshuffle or generic 3D engine redesign. |
| Package broadening | `Videra.Avalonia` does not regain transitive glTF/OBJ importer package dependencies. |
| Renderer feature breadth | No animation, skeletal/morph, lighting, shadows, environment maps, post-processing, OIT, refraction/transmission, or new material framework. |
| Platform breadth | No second public UI adapter, Wayland native embedding, WebGL/OpenGL backend, or new backend/platform family. |
| SurfaceCharts merger | Do not make `SurfaceChartView` a `VideraView` mode or push chart semantics into viewer contracts. |
| Editor/project format | Minimal reproduction export is support metadata, not a persistent scene/project/archive system. |
