# Requirements: v2.34 Beads Multi-Agent Coordination Closure

**Defined:** 2026-04-28  
**Core Value:** 跨平台 3D 渲染引擎的可靠性

## v2.34 Requirements

Requirements for this milestone. Each maps to one roadmap phase.

### Beads Service Contract

- [x] **BDS-01**: Maintainers and agents can verify that this repository uses the Docker-backed `dolt-sql-server` service at `127.0.0.1:3306` with the `Videra` database and matching Beads project id.
- [x] **BDS-02**: Repository instructions explain the minimum Beads commands for ready work, claiming, linking discovered work, closing work, and syncing Dolt state.
- [x] **BDS-03**: Beads configuration stays repo-scoped and does not point Videra at unrelated Dolt databases such as `AgentDialog`.

### Worktree Coordination

- [x] **WT-01**: Existing and future parallel phase worktrees have a documented redirect pattern so all agents share the same Beads issue truth.
- [x] **WT-02**: Maintainers can run one validation command set to confirm main and worktree contexts resolve to the same Beads database.
- [x] **WT-03**: Worktree coordination guidance makes branch/worktree isolation explicit without requiring a monolithic orchestration script.

### Issue Lifecycle and Handoff

- [x] **ISS-01**: Multi-agent work uses Beads issue lifecycle semantics for claim, in-progress status, discovered follow-up work, close reason, and owner handoff.
- [x] **ISS-02**: The milestone includes a repo-owned proof that a Beads issue can be created, claimed, closed, and observed from the Docker-backed `Videra` database.
- [x] **ISS-03**: Session closeout guidance distinguishes Git commits/pushes from Beads Dolt commits/sync so neither state channel is silently stranded.

### Validation and Guardrails

- [x] **VAL-01**: A lightweight repository validation surface checks Beads context, doctor status, worktree redirects, and Docker database metadata without mutating product code.
- [x] **VAL-02**: Guardrails prevent Beads from being documented as a replacement for GSD requirements/roadmaps, CI quality gates, release approvals, or product architecture decisions.
- [x] **VAL-03**: Documentation and tests avoid requiring Beads service startup in normal product builds or CI unless an explicit coordination validation command is invoked.

## Future Requirements

Deferred to future milestones.

### Coordination Automation

- **COORDF-01**: Add richer Beads/Dolt remote push/pull automation after the remote sync policy is agreed.
- **COORDF-02**: Add generated per-agent dashboards after the basic issue lifecycle and worktree redirect contract is stable.
- **COORDF-03**: Integrate Beads status into release-readiness evidence only after coordination validation proves stable and useful.

### Tooling

- **TOOLF-01**: Consider upgrading Beads CLI after validating schema compatibility and migration behavior on a copy of the `Videra` database.
- **TOOLF-02**: Add a Docker Compose/runbook contract only if the existing shared Docker service becomes hard to reproduce across machines.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Replacing GSD planning artifacts | Beads tracks work items; GSD remains the requirements, roadmap, and phase planning system. |
| Replacing CI or release gates | Coordination state is not build, package, native validation, benchmark, or release approval evidence. |
| Auto-starting or mutating Docker services in product commands | Beads service management stays local tooling, not part of normal builds or product runtime. |
| Migrating unrelated Dolt databases | `AgentDialog` and other databases are not Videra project state. |
| Broad multi-agent orchestration framework | v2.34 keeps coordination simple: Beads issue truth, worktree redirects, and validation commands. |
| Product runtime, renderer, chart, Demo, package, feed, or release publication work | This milestone is coordination infrastructure only. |
